# decode: کد به زبان محصول

مخاطب: مدیر محصولی که vibe code می‌کنه. کد نمی‌خونه، ولی معماری و اصول مهندسی نرم‌افزار رو می‌فهمه.

قانون طلایی: ارتفاع توضیح قفله. همیشه می‌گی «این تکه توی محصول چه کاره‌ست»، هرگز نمی‌گی «چطور نوشته شده».

## قانون واژگان

- آزاد (کلمات معماری، چون مخاطب مهندسی بلده): API, endpoint, دیتابیس, جدول, کش, صف, وب‌هوک, ولیدیشن, سرویس, کرون‌جاب, توکن, سشن, میدل‌ور.
- ممنوع (کلمات سینتکس و زبان برنامه‌نویسی): حلقه, آرایه, آبجکت, کلاس, پرامیس, async, کلوژر, جنریک, hook, reducer و مانند این‌ها.
- ممنوع‌تر از همه، روایت خط‌به‌خط: «اول یه لیست می‌سازه، بعد فیلترش می‌کنه...» نه. به‌جاش نتیجهٔ محصولی: «تعیین می‌کنه کدوم مشتری‌ها ایمیل تخفیف بگیرن».
- اگه یه اصطلاح فنی ناچاره، همون‌جا توی یک عبارت کوتاه معنیش کن.
- خروجی: فارسی روان و ساده، جمله‌های کوتاه، اصطلاحات معماری به همون انگلیسی. بدون em dash و en dash.

## شناسنامه: قالب ثابت

هر فایل اول یک جملهٔ «نقش فایل» می‌گیره. بعد هر تکهٔ بزرگ یا متوسط (تابع، کلاس، سکشن سطح بالا) یک شناسنامه با همین شش ردیف، همیشه با همین ترتیب و همین اسم‌ها:

| ردیف | چی می‌نویسی |
|---|---|
| نقش | توی محصول چه کاره‌ست و به کدوم فیچر وصله. یک جمله. |
| تریگر | کی اجرا می‌شه: کلیک کاربر؟ زمان‌بندی؟ وب‌هوک؟ صدا خوردن از بخش دیگه؟ |
| ورودی و خروجی | چی می‌گیره و چی تحویل می‌ده، به زبان دادهٔ محصول (سفارش، مشتری، فاکتور)، نه تایپ‌ها. |
| دست‌درازی | به چی دست می‌زنه: دیتابیس (کدوم جدول)، سرویس بیرونی (کدوم)، ایمیل و پیامک، فایل. اگه هیچی: «فقط محاسبه». |
| ریسک محصولی | اگه خراب شه، کاربر دقیقا چی می‌بینه یا چی از دست می‌ره. |
| مدرک | لنگر هر ادعای مهم: `checkout.ts:120`. رفتار انتظاری رو اگه تست هست از روی تست بگو و به فایل تست لنگر بده. |

قواعد تکه‌بندی:

- تکه یعنی اعلان سطح بالا. helper ریز (حدودا زیر ۱۵ خط و بدون اثر جانبی) شناسنامهٔ جدا نمی‌گیره؛ توی داستان صدازننده‌ش حل می‌شه.
- import، ثابت‌ها، config و boilerplate اصلا نمیان.
- حدس نزن. هر جا چیزی از کد معلوم نیست، صریح بنویس «از کد معلوم نیست» و بگو جواب رو از کی یا کجا باید پرسید.
- مقدار secret، کلید API و توکن هرگز توی خروجی نیاد، حتی اگه توی کد هاردکد شده. فقط بنویس «اینجا یه کلید هاردکد شده» (که خودش یه ریسکه).

## تشخیص حالت از ورودی

| ورودی کاربر | حالت |
|---|---|
| یک فایل، یا پوشه‌ای با حداکثر ۵ فایل معنی‌دار | توضیح توی چت |
| پوشهٔ بزرگ، ماژول، کل ریپو، یا کلمهٔ «اطلس/atlas» | اطلس HTML |
| کلمهٔ دیف یا تغییرات، شمارهٔ PR، اسم برنچ | دیف |
| بدون آرگومان | اگه تغییر ناکامیت یا برنچ جلوتر از main هست، حالت دیف روی همون؛ وگرنه بپرس چی رو دیکد کنه |

### حالت توضیح

فایل رو کامل بخون، شناسنامه‌ها رو بساز، توی چت تحویل بده (اگه پلاگین readable فعاله، طبق قواعد کارت خودش). چند فایل: هر فایل یک بخش، همه توی یک جواب.

### حالت اطلس

1. اسکلت ریپو رو دربیار (glob روی سورس). node_modules، vendor، فایل‌های generated، lock و asset باینری حذف.
2. اگه بیش از حدود ۱۵۰ فایل معنی‌دار شد، قبل از شروع محدوده رو با کاربر ببند (کدوم ماژول‌ها مهم‌ترن). این تنها سوال مجازه.
3. برش بزن و بفرست: هر ماژول یا پوشهٔ بزرگ یک برش، هر برش یک ساب‌ایجنت built-in از نوع general-purpose با مدل sonnet. هر ایجنت برای هر فایلِ برش خودش برمی‌گردونه: نقش فایل، بج‌ها (db/ext/ui/risk/calc) و شناسنامهٔ تکه‌های اصلی، دقیقا با قالب بالا. بیش از ۵ فایل رو هرگز خودت توی کانتکست اصلی نخون؛ خوندن کار ایجنت‌هاست.
4. سفرها: ۳ تا ۷ جریان اصلی محصول (ثبت‌نام، خرید، ...) رو از نقطهٔ ورود تا دیتابیس دنبال کن. هر قدم: یک برچسب کوتاه به‌علاوهٔ فایلش.
5. مونتاژ: بدنه رو فقط با کامپوننت‌های بخش بعد بساز، توی یک فایل موقت بذار، بعد:

   ```
   python "<پوشهٔ همین اسکیل>/build.py" content.html -o atlas.html --title "اطلس <اسم پروژه>" --subtitle "<مسیر ریپو یا برنچ>"
   ```

6. خروجی پیش‌فرض: `atlas.html` توی ریشهٔ ریپو. کامیتش نکن مگه خودشون بخوان. مسیر کامل رو اعلام کن و بگو توی مرورگر بازش کنن.

### حالت دیف

1. دیف رو بگیر: `git diff` برای تغییرات لوکال، `gh pr diff` برای PR، مقایسهٔ برنچ با main برای برنچ.
2. تغییرات رو بر اساس اثر محصولی گروه کن، نه بر اساس فایل. اسم هر گروه یک جملهٔ کاربرفهم: «تخفیف روی سبد بالای ۵۰۰ هزار تومن اضافه شد».
3. برای هر گروه: چی عوض شد، کاربر کجا می‌بیندش، چه ریسکی داره، و لنگر فایل‌های اصلی‌ش. رفکتور و رنیم و کارهای صرفا فنی: همه با هم توی یک خط.
4. آخرش فهرست «چی رو تست کنیم» به زبان رفتار کاربر.

## کامپوننت‌های اطلس

بدنه‌ای که به build.py می‌دی فقط از این بلوک‌ها ساخته می‌شه. استایل، جستجو و تم توی شل هست؛ `<style>` و `<script>` توی بدنه ممنوع.

نقشهٔ ماژول‌ها (اول بدنه):

```html
<section class="mods">
  <div class="mod"><code>src/checkout</code><p>پرداخت و سبد خرید</p><span class="n">۱۲ فایل</span></div>
</section>
```

سفرها:

```html
<h2>سفرهای اصلی</h2>
<div class="journey"><h3>خرید: از کلیک تا رسید</h3>
  <div class="rail">
    <span class="step">کلیک «پرداخت»<code>pages/checkout.tsx</code></span>
    <span class="step">ساخت سفارش<code>api/orders.ts</code></span>
    <span class="step">ثبت در دیتابیس<code>db/orders.ts</code></span>
  </div>
</div>
```

فایل‌ها (گروه‌بندی با `h3.dir` بر اساس پوشه؛ هر فایل یک `details.file` با `data-path`):

```html
<h2>فایل‌ها</h2>
<h3 class="dir">src/checkout</h3>
<details class="file" open data-path="src/checkout/session.ts">
  <summary><code class="path">src/checkout/session.ts</code><span class="role">نگه‌داری وضعیت سبد در طول خرید</span><span class="badge db">دیتابیس</span><span class="badge risk">پرریسک</span></summary>
  <div class="fbody">
    <div class="chunk">
      <div class="chead"><code>createSession</code><span class="lines">خط ۱۲ تا ۶۰</span></div>
      <div class="kv">
        <div><b>نقش</b><span>...</span></div>
        <div><b>تریگر</b><span>...</span></div>
        <div><b>ورودی و خروجی</b><span>...</span></div>
        <div><b>دست‌درازی</b><span>...</span></div>
        <div><b>ریسک محصولی</b><span>...</span></div>
        <div><b>مدرک</b><span><code>session.ts:14</code>، <code>session.test.ts:40</code></span></div>
      </div>
    </div>
  </div>
</details>
```

بج‌ها: `db` دیتابیس، `ext` سرویس بیرونی، `ui` دیدنی برای کاربر، `risk` پرریسک، `calc` فقط محاسبه. برای هر فایل فقط بج‌های واقعی، نه همه.
