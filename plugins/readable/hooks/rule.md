<readable-card-rule>
Replies render as styled cards through the `card` tool (readable-card MCP server, bundled with this plugin). When that tool is available:
- Persian or any RTL language: ALWAYS deliver the reply through it, no matter how short. Plain RTL chat text scrambles (BiDi); the card renders it correctly.
- English/LTR: use it too for conversational, explanatory, or structured answers. Skip it only when the reply is dominated by code blocks, diffs, or logs, or is a one-line status note during ongoing tool work; those stay plain text.

Call the tool exactly once, with the ENTIRE reply as the `html` argument. The call IS the reply: output nothing after it and never repeat the content as plain text, even if you suspect it did not render.

Build `html` from these blocks only (no <style>, no <script>, no wrapper div; the server template styles everything, light and dark): <h2> once as the title; <p class="lead"> muted intro; <h3> per section; <p>; <ul>/<ol>; status items <li class="ok">/<li class="no">; callouts <div class="cal tip|note|warn|danger"><div>text</div></div>; <table><thead><tbody>; chips <span class="badge ok|warn|info">; key-values <div class="kv"><div><b>k</b><span>v</span></div>...</div>; KPI cards <div class="grid c3|c2"><div class="kpi"><div class="l">label</div><div class="n">1.2M<span class="trend up">18%</span></div></div></div>; bars <div class="bars"><div class="bar"><span class="l">l</span><span class="t"><i style="width:72%"></i></span><span class="v">72%</span></div></div>; flow <div class="flow"><span class="s">step</span>...</div>; timeline <div class="tl"><div><b>t</b>text</div>...</div>; <code> around every inline path, URL, and code token; <pre><code>...</code></pre> for multiline code (renders LTR); optional closing CTA, max two, only when a natural next step exists: <div class="btns"><button class="cta" onclick="sendPrompt('the exact prompt')">label</button><button class="cta ghost" onclick="sendPrompt('...')">label</button></div>. Pick the lightest structure that fits: a short answer is one or two plain <p>, and no component is ever used just because it exists. One bidi caveat: an RTL line that must START with a Latin token needs &rlm; prefixed (or lead with an RTL word).

If the `card` tool is NOT available in this session: English replies stay plain text; Persian/RTL replies are written structurally BiDi-safe (every line starts with an RTL word; Latin tokens sit between RTL words with no attached punctuation; no hyphenated numeric ranges). If an inline HTML widget tool (mcp__visualize__show_widget) exists, prefer delivering Persian/RTL replies through it instead, as one self-contained card (this plugin ships the kit for that path in hooks/rule-inline.md).

Scope: replies rendered TO the user in chat. Text ghost-written AS the user (emails, messages meant to be forwarded) and generated files stay raw. Trivial one-line English confirmations need no card.
</readable-card-rule>
