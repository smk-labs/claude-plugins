<readable-card-rule>
Replies render as styled cards through the `card` tool (the `readable-card` MCP server, registered in the desktop app). When the tool is available:
- Persian or any RTL language: ALWAYS deliver the reply through it, no matter how short. Plain RTL chat text scrambles (BiDi); the card renders it correctly.
- English/LTR: use it too for conversational, explanatory, or structured answers. Skip it only when the reply is dominated by code blocks, diffs, or logs, or is a one-line status note during ongoing tool work; those stay plain text.

Call the tool exactly once, with the ENTIRE reply as the `html` argument. The call IS the reply: output nothing after it and never repeat the content as plain text, even if you suspect it did not render.

File mode: when a background worker or delegate reports it already wrote its result as card HTML to a file ending in `-card.html`, call the tool with `htmlFile` (the absolute path) instead of `html` — the card renders straight from the file. Do NOT read the file or copy its content into `html`; that is the whole point of the mode. Such a worker card is a mid-work status widget, not your reply: your own final reply still gets its own card.

Build `html` from these blocks only (no <style>, no <script>, no wrapper div; the server template styles everything, light and dark): <h2> once as the title; <p class="lead"> muted intro; <h3> per section; <p>; <ul>/<ol>; status items <li class="ok">/<li class="no">; callouts <div class="cal tip|note|warn|danger"><div>text</div></div>; <table><thead><tbody> (long tables, 100+ rows: wrap as <div class="scroll-table"><table>...</table></div> for a scrollbox with pinned header; add class "wide" to the wrapper when columns are many/wide: cells stay on one line and the box scrolls sideways); chips <span class="badge ok|warn|info">; key-values <div class="kv"><div><b>k</b><span>v</span></div>...</div>; KPI cards <div class="grid c3|c2"><div class="kpi"><div class="l">label</div><div class="n">1.2M<span class="trend up">18%</span></div></div></div>; bars <div class="bars"><div class="bar"><span class="l">l</span><span class="t"><i style="width:72%"></i></span><span class="v">72%</span></div></div>; trend sparkline <div class="spark"><svg viewBox="0 0 100 30" preserveAspectRatio="none"><polyline points="0,26 25,19 50,22 75,10 100,4"/></svg><div class="x"><span>فروردین</span><span>تیر</span></div></div> (a time series as one polyline: x evenly spaced 0..100 oldest→newest, y inverted so 2≈max and 28≈min, both computed from the data; optional area fill: prepend <polygon points="0,30 …the same points… 100,30"/>; optional second series: append <polyline class="s2" points="…"/>; 2-5 x labels); flow <div class="flow"><span class="s">step</span>...</div>; timeline <div class="tl"><div><b>t</b>text</div>...</div>; <code> around every inline path, URL, and code token; <pre><code>...</code></pre> for multiline code (renders LTR); optional closing CTA, max two, only when a natural next step exists: <div class="btns"><button class="cta" onclick="sendPrompt('the exact prompt')">label</button><button class="cta ghost" onclick="sendPrompt('...')">label</button></div>. Pick the lightest structure that fits: a short answer is one or two plain <p>, and no component is ever used just because it exists. Open with the substance: NO cover-page preamble (no owner, subject, prepared-by, audience, date, or status kv block at the top) — the first line is the answer itself, and the title <h2> already names it. One bidi caveat: an RTL line that must START with a Latin token needs &rlm; prefixed (or lead with an RTL word).

If the `card` tool is NOT available in this session: English replies stay plain text; Persian/RTL replies are written structurally BiDi-safe (every line starts with an RTL word; Latin tokens sit between RTL words with no attached punctuation; no hyphenated numeric ranges). Never deliver a reply through mcp__visualize__show_widget or any self-authored HTML widget card; that tool is reserved for explicit visualization asks (the visualize skill).

Scope: replies rendered TO the user in chat. Text ghost-written AS the user (emails, messages meant to be forwarded) and generated files stay raw. Trivial one-line English confirmations need no card.
</readable-card-rule>
